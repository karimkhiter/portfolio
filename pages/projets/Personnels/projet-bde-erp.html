Contexte et Mise en Situation

Digitaliser un commerce étudiant de A à Z

Le Bureau des Étudiants du département Science des Données (IUT de Metz) gérait la vente de snacks et de boissons de manière archaïque : cahiers de brouillon, fichiers Excel instables, et calculs mentaux propices aux erreurs. Les pertes financières dues aux "ardoises" (dettes étudiantes non remboursées) et aux produits périmés devenaient problématiques.

Ma mission (en tant que Président et développeur) : Concevoir, développer et déployer un système d'information complet, sécurisé et multi-supports (utilisable aussi bien sur le PC du local que sur le smartphone personnel des membres du bureau lors d'événements extérieurs).

Modélisation & Architecture de la Base de Données

La pierre angulaire d'un ERP réside dans la solidité de son modèle relationnel. Pour répondre aux besoins de traçabilité, j'ai conçu un schéma SQL normalisé.

Séparer les "Produits" des "Lots"

L'une des erreurs courantes est d'associer la quantité directement au produit. Dans mon modèle, la table produits contient les méta-données (nom, prix, type). Le stock physique, lui, est géré dans une table enfant lots. Cela permet d'avoir plusieurs dates de péremption pour un même article.

Impact métier : Lorsqu'un Kinder Bueno est vendu, le backend interroge la table lots et décrémente automatiquement le lot dont la DLUO (Date Limite) est la plus proche, respectant ainsi le standard industriel FIFO (First In, First Out).

Traçabilité Financière : Prix vs Paiement

Pour gérer intelligemment la comptabilité étudiante, j'ai implémenté les colonnes prix_final (la valeur théorique du panier), montant_paye (ce que l'étudiant a réellement donné en cas de pourboire) et un booléen est_paye (pour la gestion des ardoises/dettes).

Plongée dans le Code : Les Défis Techniques

1. Le Moteur de Formules & Réductions (JavaScript)

La difficulté côté client était de calculer le total du panier de manière dynamique sans surcharger le serveur. Un étudiant pouvait être éligible à plusieurs formules (ex: 2 snacks + 1 boisson = -0.15€). L'algorithme développé trie d'abord les formules par rentabilité, puis les applique en boucle tant que les articles requis sont présents dans le panier.

Extrait de caisse.js — Application des formules en cascade :

// TRI CRITIQUE : On trie les formules par montant de réduction décroissant
if(Array.isArray(formulesGlobal)) {
    formulesGlobal.sort((a, b) => parseFloat(b.montant_reduction) - parseFloat(a.montant_reduction));
}

// Application récursive
formulesGlobal.forEach(formule => {
    const reqSnack = parseInt(formule.nb_snacks_requis || 0);
    const reqBoisson = parseInt(formule.nb_boissons_requis || 0);
    if (reqSnack === 0 && reqBoisson === 0) return;

    // Tant qu'on a assez d'articles pour déclencher la plus grosse formule
    while (snacksDispo >= reqSnack && boissonsDispo >= reqBoisson) {
        totalReductionFormules += parseFloat(formule.montant_reduction);
        snacksDispo -= reqSnack;
        boissonsDispo -= reqBoisson;
    }
});

2. Transactions ACID (PHP / SQL)

Lorsqu'une commande est validée, le back-end doit effectuer plusieurs requêtes (création commande, lignes d'achats, décrémentation des lots, usage d'un code promo). Si l'une des requêtes échoue (ex: panne réseau), la base de données risque d'être corrompue. Pour éviter cela, j'ai utilisé les Transactions PDO (beginTransaction, commit, rollBack).

Extrait de api/save_order.php — Intégrité des données :

try {
    $pdo->beginTransaction(); // Début de la zone sécurisée

    $stmt->execute([$input['paiement'], $input['total']]);

    $pdo->commit(); // Validation définitive
} catch (Exception $e) {

}

3. Le Système Actif d'Ardoises (Dettes)

Fonctionnalité critique demandée par le BDE : autoriser un étudiant à prendre un snack et payer plus tard. Le système isole ces commandes via est_paye = 0. L'interface affiche un encart dynamique rouge en haut de l'historique regroupant les dettes par étudiant, avec un bouton permettant, via une requête AJAX, de solder toutes ses commandes d'un seul clic.

Expérience Utilisateur (UX) & Responsive Mobile-First

Transformation des Tableaux en "Cards" (CSS)

Sur un écran d'ordinateur, l'administration s'affiche sous forme de tableaux (HTML <table>). Sur smartphone, pour éviter les débordements horizontaux hideux, j'ai développé un CSS spécifique utilisant les attributs data-label. Ce script CSS "détruit" la structure rigide du tableau pour transformer chaque ligne en une véritable carte d'information (Flexbox en colonne), offrant une ergonomie d'application native.

@media screen and (max-width: 900px) {
    /* Les cases du tableau deviennent des colonnes Flex */
    .admin-table td {
        display: flex !important;
        flex-direction: column !important;
    }

    /* Génération du titre de la donnée dynamiquement via CSS */
    .admin-table td::before {
        content: attr(data-label) !important;
        font-weight: 800 !important;
        color: var(--primary) !important;
        text-transform: uppercase !important;
    }
}

Le système intègre également des feedbacks visuels asynchrones : les produits en rupture se grisent automatiquement, les badges de notification du panier sur mobile s'animent via CSS keyframes lors de l'ajout d'un article, et la barre de navigation glisse horizontalement pour faciliter l'accès au pouce.
